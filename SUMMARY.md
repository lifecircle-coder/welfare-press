# 📑 프로젝트 아키텍처 및 안전 작업 가이드 (SUMMARY.md)

이 문서는 프로젝트의 현재 상태를 기록하고, 향후 수정 작업 시 **시스템의 안정성을 100% 보장하기 위한 엄격한 규칙**을 정의합니다. 새로운 대화창에서 작업을 시작할 때 이 파일을 통해 컨텍스트를 즉시 파악할 수 있도록 합니다.

## 🚨 절대 준수 규칙 (Absolute Rules)

1. 모든 업무는 총괄 PM인 제이한테 지시하고, 제이가 업무에 맞는 팀원에게 업무를 하달하는 방식으로 토큰 소모를 최소화한다.
2. 내가 요청한 업무 외 다른 것은 절대 하지 않는다.
3. 작업을 진행할 때 다른 잘 작동하는 것을 절대 건드리지 않고, 작업하지도 않는다. 이유는 수정을 하면 다른 것까지 건드려서 잘 작동했던 곳에서도 오류가 발생하기 때문
4. 작업 전 모든 전수조사를 통해 작업하지 않은 곳에서 발생하는 오류 발생률을 0%로 한다.
5. 요청한 업무에 대해서 반복적인 오류 발생을 최소화하고 한번에 요청사항을 완료한다.
6. 작업 전 작업자들은 SUMMARY.md 파일의 절대규칙을 다시한번 확인하고 꼭 이 절대규칙을 준수하면서 작업한다.
7. 업무 진행 프로세스는 내가 제이에게 업무를 지시하면 제이는 케이에게 아키텍트를 만들고 그것을 내가 승인하면 제이는 케이가 만든 아키텍트를 보고 맞는 팀원에게 업무를 할당하고 작업이 완료되면 큐는 케이가 작성한 아키텍트처럼 구현이 되었는지 실제 서비스에 접속해서 확인한 후 제이에게 보고한 후 제이가 최종 확인 후 나에게 보고한다.
8. 시간이 오래 걸리거나 내가 직접 하면 더 빠르게 진행되는 일은 나에게 바로 요청한다. 예를들어 깃허브 인증이나, supabase 스크립트 요청 등. 내가 직접해서 빠르게 진행될 수 있는 업무면 바로바로 요청한다.
9. 내가 요청한 작업은 내가 **다음 작업으로 넘어갈게** 등 내가 완료되었다고 말하기 전까지는 절대 제이 스스로 판단해서 완료되었다고 보고하지 않는다.
10. 내가 요청한 작업이 완료될 때마다 내가 말하지 않아도 SUMMARY.md 파일에 업데이트를 해서 새로운 대화창에서도 바로 다음 작업을 할 수 있도록 꼼꼼하게 작성한다. 특히 내가 요청한 내용이나, 이 요청한 내용을 케이가 작성한 아키텍트 등을 꼼꼼히 작성해서, 헷갈리거나 누락이 되는 일이 없도록 한다.
11. 내가 요청한 작업이 완료될 때마다 작업한 내용을 백업해서 내가 **언제로 롤백해** 라고 하면 롤백을 할 수 있게 백업관리를 철저히 한다.
12. **'로컬서버 테스트 개발용 프로젝트 supabase에 생성해'** 라고 요청하면, 실서버와 완전히 격리된 별도의 개발용(Dev) Supabase 프로젝트를 생성하고 환경 설정(.env.local)까지 완료한다. (진행 전 SUMMARY.md 및 PRD 확인 필수)

## 🛡️ 안전한 작업을 위한 3대 원칙 (Safety Protocols)

### 1. "무영향성 검증" (No-Impact Verification)
- **사전 분석**: 코드를 수정하기 전, 해당 변경이 기존 기능(특히 발행일 보존, 정렬 성능)에 영향을 주지 않는지 논리적으로 분석하여 보고합니다.
- **대안 제시**: 위험 요소가 발견되면 숨기지 않고 안전한 작업 방식을 우선 제안합니다.

### 2. 큐(QA)의 엄격한 전수조사 (Regression Testing)
- **수정 확인**: 요청한 사항이 정확히 반영되었는지 테스트합니다.
- **기존 기능 유지**: 메인 페이지, 상세 페이지, 관리자 목록이 정상인지 전수 조사합니다.
- **고정값 검증**: 이미 해결된 '발행일(date) 보존' 및 '인덱스 성능'이 그대로 유지되는지 확인합니다.

### 3. '롤백 지점' 명시 및 스냅샷 보존 (Rollback Strategy)
- **작업 전 백업**: 수정 전 완벽한 상태를 Git 커밋 및 로컬 스냅샷으로 보존합니다.
- **즉각 원복**: 0.1%라도 이상이 생기면 1초 만에 가장 완벽했던 이전 상태로 되돌릴 것을 약속합니다.

### 4. "실서버 직접 검증" (Live-Server Verification)
- **완료 조건**: 단순히 코드가 수정되었거나 로컬 테스트가 통과했다고 해서 완료로 판단하지 않습니다.
- **직접 테스트**: 모든 수정 사항은 푸시 후 큐(QA)가 **직접 실서버 관리자 페이지 및 프론트에 접속하여 사용자의 요청 사항이 100% 반영되었는지 눈으로 확인**해야 합니다.
- **보고 기준**: 사용자의 요청과 실서버의 구동 결과가 일치할 때만 제이(PM)가 최종 확인 후 보고합니다.

## 🚀 워크플로우 및 도구 활용
1. **Branch 활용**: `main` 브랜치는 항상 동작하는 상태로 유지하며, 모든 수정은 `feat/` 또는 `fix/` 브랜치에서 진행 후 검증된 것만 합칩니다.
2. **Atomic Updates**: 한 번에 하나씩, 가장 중요한 것부터 고치고 완벽하게 작동하는지 확인 후 다음 단계로 넘어갑니다.
3. **Implementation Plan**: 작업 전 케이(Architect)의 '구현 계획서'를 통해 예상 영향도와 절차를 미리 공유하고 승인을 득합니다.
4. **뇌피셜 코딩 금지 및 MCP 무기고 활용 (Zero-Hallucination Policy)**
**프레임워크 문법**: Next.js 및 Supabase 관련 코드를 작성/수정하기 전, 반드시 Context7 MCP를 호출하여 최신 공식 문서와 API 레퍼런스를 먼저 확인한다.
**DB 스키마 보호**: 백엔드 수정 시 원시 SQL 문을 맹목적으로 실행하지 말고, Supabase MCP를 통해 읽기 전용(Safe type) 쿼리로 현재 스키마를 먼저 확인한다.
**UI/UX 수정**: 시각적인 디자인 요소 추가 시 백엔드 로직을 건드리지 말고 Stitch MCP를 호출하여 시각적 렌더링 로직을 비즈니스 로직과 철저히 격리한다.
5. **결론 도출 전 검증 (Verify-Before-Conclude)**
모든 버그 수정 및 새로운 기능 추가 지시를 받으면, 코드를 즉시 토해내지 않는다.
반드시 Sequential Thinking 스킬을 가동하여 수정 사항이 기존 의존성(발행일 보존 등)에 파급 효과(Ripple effect)를 일으키지 않는지 교차 검증하고 가설을 세운다.

## 🔧 핵심 아키텍처 및 기술 규칙
- **Tech Stack**: Next.js 14+ (App Router), Supabase (PostgreSQL), TypeScript, Tailwind CSS.
- **Date Integrity (중요)**: 
    - `date` field는 기사 발행 시점의 고유값으로, 기사 수정 시 절대 변하지 않아야 합니다.
    - 관리자 UI에서 발행일은 `readOnly` 상태를 유지합니다.
- **Session Isolation & Auth Integrity (인증 무결성)**:
    - 홈페이지(Public)와 관리자(Admin) 세션의 충돌을 방지하기 위해 `lib/services.ts`는 반드시 '클라이언트 주입(Client Injection)' 패턴을 사용합니다.
    - **VisitorTracker 격리**: 공용 세션을 사용하는 방문자 추적기(`VisitorTracker.tsx`)는 관리자 경로(`/admin`)에 진입 시 즉시 실행을 중단하여 관리자 인증 세션 오염을 원천 차단합니다.
    - 관리자 페이지 및 기능 호출 시에는 항상 명시적으로 `adminSupabase` 클라이언트를 주입하여 공용 세션과의 오염을 차단합니다.
- **Performance**: `created_at` 인덱스(`idx_articles_created_at`)가 생성되어 있으며, 대량 Base64 이미지 본문 삽입은 지양합니다.
- **UI Standard**: 
    - **Max Width**: 홈페이지의 서비스 가시성과 안정감을 위해 **최대 가로폭을 1280px로 제한**합니다. (포털 사이트 Naver 표준 준수)
    - **반응형**: 1280px 미만 해상도에서는 콘텐츠가 화면 크기에 맞춰 자연스럽게 줄어드는 **유동형 반응형(Fluid)** 방식을 유지합니다.
    - 모든 날짜와 시간은 초 단위(`toLocaleString('ko-KR')`)까지 명확히 표시합니다.
    - 본문 에디터는 **텍스트/이미지 정렬(Align)**, **폰트 크기(Size)**, **텍스트 색상(Color)** 조절 기능을 지원합니다.
    - 실시간 렌더링 및 스타일 보존을 위해 **인라인 스타일(Inline Style)** 방식을 사용합니다.
    - 기사 저장/수정 시 **알림창(alert)**을 통해 사용자에게 작업 성공 여부를 알립니다.
